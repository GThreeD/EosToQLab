# Contributing to EosToQLab

Thank you for helping improve EosToQLab. Contributions may include bug reports, parser fixtures, documentation, tests, and code changes.

By participating, you agree to follow the [Code of Conduct](CODE_OF_CONDUCT.md).

## Before opening an issue

- Search existing issues to avoid duplicates.
- Use the dedicated parser compatibility form for `.esf2` or `.esf3d` import problems.
- Remove passcodes, network details, personal information, production names, and confidential show content from logs and fixtures.
- Do not publish security vulnerabilities in a normal issue; follow [SECURITY.md](SECURITY.md).

## Development prerequisites

The application targets macOS and uses .NET 10, .NET MAUI, and Blazor Hybrid.

Typical prerequisites are:

- macOS with a compatible Xcode version;
- the .NET SDK version selected by `global.json`;
- the .NET MAUI workload;
- QLab 5 for live integration checks;
- ETC Eos or an Eos export only when creating a new compatibility fixture.

Install the MAUI workload and restore the solution:

```bash
dotnet workload install maui
dotnet restore EosToQLab.slnx
```

## Build and test

Run the complete automated test suite:

```bash
dotnet test tests/EosToQLab.Tests/EosToQLab.Tests.csproj -c Release
```

Run the same line-coverage gate used by CI:

```bash
dotnet test tests/EosToQLab.Tests/EosToQLab.Tests.csproj -c Release \
  -p:CollectCoverage=true \
  -p:CoverletOutputFormat=cobertura \
  -p:Threshold=90 \
  -p:ThresholdType=line \
  -p:ThresholdStat=total \
  '-p:Include=[EosToQLab.Core]*,[EosToQLab.Infrastructure]*'
```

Build the Mac Catalyst application:

```bash
dotnet build src/EosToQLab.Application/EosToQLab.Application.csproj \
  -f net10.0-maccatalyst
```

QLab and Eos are not required for deterministic unit tests. Live integration behavior should be checked manually when a change affects OSC communication, workspace selection, patch selection, cue creation, rollback, or saving.

## Project expectations

Keep changes focused and preserve the existing architecture:

- Core contains stable models, contracts, diagnostics, and application-facing behavior.
- Infrastructure contains CSV, EOS show-archive, and QLab OSC implementation details.
- Application contains the MAUI Blazor Hybrid user interface and dependency registration.
- Source importers map into the common `EosCue` model and must not depend on QLab.
- QLab services must not depend on CSV, ESF2, or ESF3D parser details.

Avoid unrelated formatting or broad refactoring in a bug-fix pull request.

## Parser fixtures

EOS show archives may contain confidential production data. Commit only minimal, artificial, or fully sanitized fixtures.

The fixture tree is organized by intent:

```text
tests/EosToQLab.Tests/Fixtures/EosShowArchive/
├── Compatibility/
└── Regression/
```

Each fixture case is a directory containing:

- exactly one `.esf2` or `.esf3d` archive;
- one `expected.json` contract;
- optional documentation.

Use `Compatibility` only for a complete parser contract representing an EOS format/version pair supported by the application. Use `Regression` for a focused bug reproduction. Generated byte sequences for isolated decoder branches belong in unit-test builders rather than committed archive folders.

Never replace an existing compatibility fixture when adding support for a new EOS version. Add a new directory so backward compatibility remains tested.

## Tests

A production change should include tests that fail before the change and pass afterward.

Parser changes should verify, as applicable:

- cue number and list number;
- label, notes, and scene text;
- Follow and Hang values;
- fractional and high cue numbers;
- diagnostic codes and warning behavior;
- `.esf2` and `.esf3d` behavior;
- false-positive records that must not become cues.

Changes to QLab import behavior should cover planning, protocol mapping, failure handling, and rollback without requiring a live QLab instance where practical.

## Pull requests

Before opening a pull request:

1. rebase or merge the latest `main` as appropriate;
2. run the complete test suite;
3. run a Mac Catalyst build when the application project changed;
4. confirm no credentials, passcodes, private show data, build output, or generated user state is included;
5. describe user-visible behavior and remaining limitations;
6. update documentation when behavior or architecture changes.

Small, reviewable pull requests are preferred. Draft pull requests are welcome for early architectural feedback.

## Commit messages

Use concise imperative messages that explain the change, for example:

```text
Fix Hang decoding for EOS 3.3.5 archives
Add ESF2 compatibility fixture discovery
Document QLab rollback behavior
```
