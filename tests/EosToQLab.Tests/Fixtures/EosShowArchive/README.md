# EOS show archive fixtures

The fixture tree separates released compatibility evidence from parser regressions:

- `Compatibility/<case>/` contains one `.esf2` or `.esf3d` archive, one `expected.json` contract, and optional
  documentation. These directories are the only input used to generate the runtime compatibility catalog.
- `Regression/<case>/` contains one archive and one contract for a previously observed parser defect or an older
  supported layout. Regression cases are tested but never advertised as additional compatible versions.

A case is defined by its directory, not by a known archive filename. The test discovery scans all case directories
recursively and requires exactly one `.esf2` or `.esf3d` archive plus one `expected.json` in each case directory. Adding
a regression therefore requires no C# test-code change.

A compatibility `expected.json` must contain an `archive` object whose `format` and `version` exactly match the archive's `version.json`. The build fails when either side is missing or disagrees.

Generated byte sequences for isolated decoder branches live directly in unit tests and test builders. They do not need a
third archive-fixture category.

When ETC releases a new Eos version:

1. Create a minimal artificial show that exercises labels, notes, scene text, cue-number fractions, Follow, and Hang.
2. Export it as `.esf2` or `.esf3d`.
3. Add a new directory below `Compatibility/` without replacing older fixtures.
4. Put exactly one archive and one `expected.json` contract in that directory.
5. Record the independently known expected cues and archive identity in `expected.json`.
6. Run the full test suite. A published package is produced only after every discovered compatibility and regression
   contract passes.
