# Testing strategy

## Why xUnit replaces the self-test console

The original self-test executable was useful while the project was small and dependency-free, but it had several
structural limitations:

- tests were manually registered in one list;
- one process returned only pass/fail text instead of standard test results;
- IDE discovery, filtering, fixtures, theories, and per-test diagnostics were unavailable;
- code coverage could be reported but not reliably enforced per assembly;
- a new production class could be added without any test convention noticing.

The repository now uses xUnit with `Microsoft.NET.Test.Sdk`. Coverlet produces Cobertura output and enforces a 100% line
threshold for Core and Infrastructure. Each production type has a dedicated `<TypeName>Tests.cs` file. The architecture
test is the guardrail, while normal behavior tests remain explicit and readable.

## Coverage scope

The 100% line gate covers:

- `EosToQLab.Core`;
- `EosToQLab.Infrastructure`.

These assemblies contain parsing, mapping, planning, OSC protocol, transport, workflow, rollback, diagnostics, and state
logic. The gate includes executable auto-property accessors as well as ordinary methods; generated compiler and
framework code remains outside the two covered assemblies.

The Mac Catalyst application shell is verified by compilation and manual UI/integration tests. Generated XAML, Apple
platform entry points, and `SecureStorage` itself are not counted as business-code unit coverage.

## One test file per production type

The convention is:

```text
src/.../QLabOscSession.cs
Tests/.../QLabOscSessionTests.cs
```

`ProductionTypeTestConventionTests` scans both production assemblies and source files. CI fails when either the
`<TypeName>Tests` class or the `<TypeName>Tests.cs` file is missing. Interfaces, enums, records, record structs, static
classes, internal protocol helpers, and exception types are included in the convention.

## EOS show archives without a live Eos dependency

The parser does not require an installed or running Eos instance in CI. Archive fixtures are divided by intent:

```text
tests/EosToQLab.Tests/Fixtures/EosShowArchive/
├── Compatibility/
│   └── <case>/
│       ├── one .esf2 or .esf3d archive
│       ├── expected.json
│       └── optional README.md
└── Regression/
    └── <case>/
        ├── one .esf2 or .esf3d archive
        └── expected.json
```

A case is identified by its directory. Tests recursively discover every case and do not depend on a particular archive
filename or a hardcoded list of regression names. Discovery fails when a case does not contain exactly one show archive
and exactly one `expected.json` contract.

Only `Compatibility/` is evidence that an archive identity is covered by a released build. `Regression/` contains
previously observed parser defects and older layouts, but its identities are never added to the runtime compatibility
catalog. Generated byte sequences for isolated decoder branches remain in unit tests and fixture builders rather than a
third archive-fixture category.

Each compatibility contract declares the identity independently in `expected.json`:

```json
{
  "archive": {
    "format": "{...}",
    "version": "3.3.5.69"
  },
  "cues": []
}
```

The archive's own `version.json` must contain the same `Format` and `Version`. The build fails when the contract is
missing, the archive manifest is unreadable, or the values disagree. `Format` and `Version` are treated as an opaque
pair; the project does not assume that the GUID changes for every schema change.

### Build-generated runtime catalog

`EosToQLab.CompatibilityCatalogGenerator` scans only discovered cases below
`Fixtures/EosShowArchive/Compatibility`. Before Infrastructure resources are prepared, MSBuild generates
`eos-show-archive-tested-archives.json` into `obj` and embeds it in `EosToQLab.Infrastructure`. No tested-version list
is
maintained by hand in production code.

The same discovered compatibility cases feed the parameterized parser-contract test. The release workflow runs all
compatibility and regression contracts before either macOS package is published. Therefore an identity can reach the
embedded runtime catalog only from a committed compatibility case, and a package is released only after the full test
job passes.

Tests never write production artifacts. Catalog generation is a deterministic build step, not an xUnit side effect.

### Adding a new Eos version

1. Create a minimal artificial show containing labels, notes, scene text, fractional cue numbers, Follow, and Hang.
2. Export one `.esf2` or `.esf3d` archive.
3. Add a new case directory under `Compatibility/`; never replace an older contract.
4. Put exactly one archive and one `expected.json` in that directory. The archive filename is unrestricted.
5. Record the independently known expected cues and archive identity in `expected.json`.
6. Run the complete xUnit suite and inspect the generated catalog artifact.
7. Put bug-specific archives under `Regression/`; adding the directory is sufficient and requires no C# test change.

## Live integration boundaries

Unit tests emulate QLab replies and exercise localhost TCP framing, but they do not replace a live QLab smoke test. QLab
version changes, OSC permissions, actual Network Patch menus, macOS drag-and-drop, and code signing remain integration
concerns covered by the macOS build and the manual checklist.
